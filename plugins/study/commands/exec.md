# Study — Exec (학습 실행)

status가 `ready`인 학습 계획서를 기반으로 5단계 구조화된 심화 학습을 실행한다. 각 Phase의 결과물을 `study/docs/{id}-{slug}-notes.md`에 누적 기록한다.

## 트리거

- `/study:exec` — ready 상태인 학습 계획서 목록을 보여주고 선택
- `/study:exec 1` — 1번 주제 학습을 처음부터 실행
- `/study:exec 1 --phase 3` — 1번 주제의 Phase 3부터 실행
- `/study:exec 1 --resume` — 1번 주제의 마지막 중단 지점부터 재개
- 자연어: "스터디 시작", "공부 시작하자", "학습 실행", "1번 주제 공부하자"

## 실행 단계

### 0. 준비

1. `study/docs/` 디렉토리를 스캔 (Glob 도구)
2. 각 학습 계획서의 frontmatter status 확인 (Read 도구)

- **번호가 주어진 경우**: 해당 계획서를 열고 status 확인
- **번호가 없는 경우**: `ready` 또는 `in-progress` 상태인 계획서 목록을 표시하고 선택 요청

```
| # | 주제 | 상태 | 난이도 | 진행률 |
|---|------|------|--------|--------|
| 1 | AWS 네트워크 인프라 | ready | intermediate | - |
| 2 | Kafka 내부 동작 원리 | in-progress | advanced | Phase 2/5 |
```

**status가 ready/in-progress가 아닌 경우**:
- `draft`: `/study:init {번호}` 후 `/study:bs {번호}` 안내
- `refined`: `/study:bs {번호}` 로 추가 브레인스토밍 안내
- `completed`: 이미 학습 완료됨을 알림, 재학습 여부 확인

### 1. 학습 노트 초기화

학습 계획서를 읽고 노트 파일을 생성/로드한다.

**노트 파일 경로**: `study/docs/{id}-{slug}-notes.md`

**노트 템플릿** (최초 생성 시):

```markdown
---
id: {번호}
title: {한글 제목} — Study Notes
plan: {id}-{slug}.md
started: {YYYY-MM-DD}
last-phase: 0
status: in-progress
---

# {한글 제목} — Study Notes

<!-- 각 Phase 완료 시 자동으로 내용이 추가됨 -->
```

**`--resume` 플래그**: 기존 노트 파일의 `last-phase`를 읽어 다음 Phase부터 재개.

학습 계획서 frontmatter를 `status: in-progress`로 변경하고, `study/temp.md` 학습 현황 표도 동기화.

## Phase 1: 기본 개념 (Core Concepts)

**목표**: 주제의 핵심 용어, 기초 이론, 동작 원리를 정리한다.

### 실행 방식

1. 학습 계획서의 `Phase 1: 기본 개념` 섹션의 소주제 목록을 읽는다
2. 각 소주제에 대해:
   - **핵심 정의** — 한 문장으로 정의
   - **동작 원리** — 어떻게 작동하는지 (다이어그램 텍스트 포함 가능)
   - **핵심 포인트** — 꼭 기억해야 할 2-3가지
   - **비유/멘탈 모델** — 이해를 돕는 비유 (적절한 경우)
3. 소주제 간 관계를 정리 (의존 관계, 상호작용)

### 작성 원칙

- **간결하고 정확하게** — 교과서가 아닌 핵심 노트
- **사용자 수준 맞춤** — 이미 아는 기초는 한 줄 리마인더로, 모르는 부분은 상세히
- **실무 관점 우선** — 학술적 정의보다 "실무에서 왜 중요한지"를 강조
- 필요시 context7 MCP를 통해 최신 공식 문서를 참조

### 노트 기록

Phase 1 완료 시 노트 파일에 추가:

```markdown
## Phase 1: 기본 개념
> 학습일: {YYYY-MM-DD}

### {소주제 1}
...

### {소주제 N}
...

### 개념 관계도
...
```

`last-phase: 1`로 업데이트.

### Phase 전환

- 사용자에게 Phase 1 요약을 보여주고, 이해도를 확인
- "Phase 2로 넘어갈까요?" — 사용자 승인 후 진행
- 사용자가 특정 개념에 대해 추가 질문하면 즉석 답변 후 노트에 반영

## Phase 2: 심화 (Deep Dive)

**목표**: 내부 구조, 성능 특성, 트레이드오프, 실전 패턴을 깊이 학습한다.

### 실행 방식

1. 학습 계획서의 `Phase 2: 심화` 섹션의 소주제 목록을 읽는다
2. 각 소주제에 대해:
   - **내부 동작** — Phase 1보다 한 단계 아래의 구현 레벨
   - **트레이드오프** — 장점 vs 단점, 언제 쓰고 언제 쓰지 않는지
   - **성능 특성** — 시간/공간 복잡도, 병목 포인트 (해당 시)
   - **실전 패턴** — 프로덕션에서 흔히 사용하는 구성/패턴
   - **안티패턴** — 흔한 실수, 왜 문제인지
3. Phase 1 개념과의 연결고리를 명시

### 작성 원칙

- **"왜"에 집중** — 동작 방식보다 설계 이유
- **비교 분석** — 유사 기술/패턴과의 차이점 (예: ALB vs NLB, Kafka vs RabbitMQ)
- **수치 근거** — 가능하면 구체적 수치 (처리량, 지연 시간 등)
- 필요시 context7 MCP를 통해 최신 공식 문서를 참조

### 노트 기록

Phase 2 완료 시 노트 파일에 추가, `last-phase: 2` 업데이트.

## Phase 3: 코드베이스 탐색 (Codebase Exploration)

**목표**: 현재 msa 프로젝트에서 이 주제가 어떻게 적용되어 있는지 탐색한다.

> **토큰 최소화 전략**: 전체 파일을 읽지 않고, Grep/Glob으로 위치를 특정한 뒤 해당 부분만 발췌한다.

### 실행 방식

1. **키워드 도출**: 학습 주제에서 코드베이스 검색에 사용할 키워드를 도출
   - 예: "AWS ALB" → `alb`, `load-balancer`, `health-check`, `target-group`
   - 예: "Kafka" → `KafkaTemplate`, `@KafkaListener`, `ConsumerConfig`, `ProducerConfig`

2. **탐색 순서** (토큰 효율순):
   ```
   Step 1: Glob — 파일명/경로로 관련 파일 특정
   Step 2: Grep — 키워드로 코드 위치 특정 (files_with_matches 모드)
   Step 3: Read — 특정된 위치의 해당 부분만 발췌 (offset + limit 사용)
   ```

3. **탐색 대상 우선순위**:
   - `k8s/` — 인프라 설정 (K8s manifest, ConfigMap, Secret)
   - `docs/adr/` — 관련 아키텍처 결정 기록
   - `docs/architecture/` — 아키텍처 문서
   - `*/app/src/` — 서비스 애플리케이션 코드
   - `*/domain/src/` — 도메인 코드
   - `common/` — 공유 라이브러리
   - `build.gradle.kts`, `settings.gradle.kts` — 의존성

4. **codebase-relevant: false인 경우**:
   - 코드베이스에 직접 연관이 없는 주제 (예: 순수 이론, 아직 적용 안 된 기술)
   - 간략한 스캔만 수행하고, "현재 프로젝트에는 미적용" 으로 기록
   - 대신 "만약 적용한다면 어디에?" 관점으로 Phase 4와 연결

### 작성 원칙

- **파일 경로 + 라인 번호 명시** — `product/app/src/.../KafkaProducerConfig.kt:23`
- **코드 발췌는 최소화** — 전체 클래스가 아닌 해당 패턴이 적용된 부분만
- **설정과 코드를 분리** — 인프라 설정(yaml/json)과 애플리케이션 코드를 구분하여 정리
- **ADR 참조** — 관련 ADR이 있으면 링크와 핵심 결정 사항 기록

### 노트 기록

```markdown
## Phase 3: 코드베이스 탐색
> 탐색일: {YYYY-MM-DD}

### 적용 현황 요약
| 위치 | 파일 | 적용 패턴 | 비고 |
|------|------|----------|------|
| product | `.../KafkaProducerConfig.kt` | Producer 설정 | ADR-0012 준수 |
| k8s | `k8s/infra/local/kafka/` | StatefulSet 배포 | 단일 브로커 |

### 상세 분석
#### {모듈/서비스명}
- 파일: `path:line`
- 패턴: {어떤 패턴으로 적용되어 있는지}
- 특이사항: {일반적 패턴과 다른 점, 개선 가능성}

### 관련 ADR
- ADR-{번호}: {제목} — {핵심 결정}
```

`last-phase: 3` 업데이트.

## Phase 4: 적용 가능 포인트 (Application Opportunities)

**목표**: 학습한 내용을 현재 코드베이스에 새로 적용하거나 개선할 수 있는 포인트를 찾는다.

### 실행 방식

1. Phase 2(심화)에서 학습한 패턴/안티패턴과 Phase 3(코드베이스)의 현재 상태를 교차 분석
2. 다음 관점으로 기회를 식별:
   - **미적용 패턴** — 학습한 베스트 프랙티스 중 아직 적용 안 된 것
   - **개선 가능** — 적용되어 있지만 더 나은 방식이 있는 것
   - **안티패턴 발견** — Phase 2에서 학습한 안티패턴이 코드에 존재하는 경우
   - **신규 도입 기회** — 프로젝트에 아예 없는 기술/패턴의 도입 가치 평가

3. 각 기회에 대해:
   - **현재 상태** — 지금 어떻게 되어 있는지
   - **제안** — 어떻게 바꾸면 좋은지
   - **영향도** — 변경 범위와 리스크
   - **우선순위** — 높음/중간/낮음 (비용 대비 효과)

### 작성 원칙

- **실행 가능한 제안만** — "이론적으로 가능"이 아닌 "이렇게 바꿀 수 있다"
- **ADR 필요 여부 표시** — 아키텍처 변경이 필요한 경우 명시
- **과도한 리팩토링 제안 자제** — 학습 목적이지 리팩토링 계획이 아님

### 노트 기록

```markdown
## Phase 4: 적용 가능 포인트
> 분석일: {YYYY-MM-DD}

| # | 제안 | 대상 | 영향도 | 우선순위 | ADR 필요 |
|---|------|------|--------|---------|---------|
| 1 | ... | ... | 낮음 | 높음 | N |
| 2 | ... | ... | 중간 | 중간 | Y |

### 상세
#### 1. {제안 제목}
- 현재: ...
- 제안: ...
- 근거: Phase 2 — {관련 심화 내용}
```

`last-phase: 4` 업데이트.

## Phase 5: 면접 대비 (Interview Prep)

**목표**: 이 주제에 대한 기술 면접을 대비한다. 꼬리 질문 패턴을 파악하고, 깊이 있는 답변을 준비한다.

### 실행 방식

1. **기본 질문 생성** (5-8개):
   - Phase 1-2 학습 내용에서 면접관이 물을 법한 핵심 질문
   - 난이도 순서: 기본 → 중급 → 고급

2. **꼬리 질문 트리 구성** (각 기본 질문당 2-3단계):
   ```
   Q1: "VPC란 무엇인가요?"
   ├── Q1-1: "서브넷을 public/private으로 나누는 이유는?"
   │   └── Q1-1-1: "NAT Gateway 없이 private 서브넷에서 외부 통신하는 방법은?"
   └── Q1-2: "VPC Peering과 Transit Gateway의 차이는?"
       └── Q1-2-1: "대규모 멀티 리전에서 어떤 걸 선택하고 그 이유는?"
   ```

3. **모범 답변 작성**:
   - 각 질문에 대해 **STAR 형식이 아닌 기술 답변 구조**:
     - **핵심 답변** (1-2문장) — 면접관이 원하는 직접적 답변
     - **설명** (2-3문장) — 왜 그런지, 어떻게 동작하는지
     - **실무 경험 연결** (1-2문장) — "저는 이 프로젝트에서 ~했습니다" (Phase 3-4 기반)
     - **+α** — 추가로 언급하면 좋은 포인트

4. **함정 질문 / 오해하기 쉬운 포인트**:
   - 흔히 잘못 알고 있는 개념
   - 면접관이 의도적으로 틀린 전제를 깔고 물어볼 수 있는 것
   - "~라고 생각하시는 분이 많은데, 실제로는 ~"

5. **시스템 설계 연결** (해당 시):
   - 이 주제가 시스템 설계 면접에서 어떻게 등장하는지
   - 관련 시스템 설계 시나리오 1-2개

### 작성 원칙

- **사용자 경력 수준에 맞춤** — 주니어용 기본 질문이 아닌 해당 연차에 맞는 깊이
- **실무 연결 필수** — Phase 3-4에서 찾은 코드베이스 경험을 답변에 자연스럽게 녹이기
- **단순 암기가 아닌 이해 검증** — "왜?"를 계속 파고드는 구조
- **한국어 답변 기준** — 한국 기업 면접 환경 고려, 영어 용어는 원문 병기

### 노트 기록

```markdown
## Phase 5: 면접 대비
> 작성일: {YYYY-MM-DD}

### 기본 질문 & 꼬리 질문 트리

#### Q1: {질문}
**답변**: ...

  **Q1-1**: {꼬리 질문}
  **답변**: ...

    **Q1-1-1**: {꼬리 질문}
    **답변**: ...

...

### 함정 질문 / 오해 포인트
| # | 오해 | 실제 | 근거 |
|---|------|------|------|

### 시스템 설계 연결
...
```

`last-phase: 5` 업데이트.

## 학습 완료 처리

모든 Phase가 끝나면:

1. **학습 계획서 상태 업데이트**: `status: in-progress` → `status: completed`
2. **노트 파일 마무리**:
   ```markdown
   ---
   status: completed
   completed: {YYYY-MM-DD}
   last-phase: 5
   ---
   ```
3. **temp.md 업데이트**:
   - 학습 현황 표에서 해당 행의 상태를 `completed`로 업데이트
4. **완료 리포트**:
   - 전체 학습 요약 (핵심 3-5줄)
   - Phase별 주요 발견/학습 포인트
   - 코드베이스 적용 제안 요약 (Phase 4)
   - 면접 준비도 자가 평가 기준 제시
   - 관련 주제 추가 학습 제안 (있으면)

## 중단 및 재개

- 어느 Phase에서든 사용자가 중단 가능
- 중단 시 현재 Phase까지의 내용을 노트에 저장하고 `last-phase` 업데이트
- `/study:exec {번호} --resume`로 마지막 Phase 다음부터 재개
- `/study:exec {번호} --phase N`으로 특정 Phase만 재실행 가능 (노트 해당 섹션 교체)

## 토큰 최소화 전략

Phase 3(코드베이스 탐색)에서 특히 중요:

1. **files_with_matches 우선** — Grep 결과에서 파일 경로만 먼저 확인
2. **offset + limit 활용** — Read 시 필요한 부분만 발췌
3. **Agent(Explore) 위임** — 광범위 탐색이 필요하면 Explore 에이전트에 위임하여 메인 컨텍스트 보호
4. **결과 요약 기록** — 코드 전문이 아닌 패턴 요약을 노트에 기록

## 에러 처리

- 학습 계획서가 없는 번호 지정 시: `/study:init {번호}` 안내
- ready/in-progress 상태가 아닌 계획서 지정 시: 현재 상태와 필요한 다음 단계 안내
- 코드베이스 탐색에서 관련 코드가 없는 경우: "미적용 주제"로 기록, Phase 4로 바로 전환
