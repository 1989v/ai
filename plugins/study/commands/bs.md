# Study — Brainstorm (학습 방향 브레인스토밍)

이미 생성된 학습 계획서를 사용자와 대화형으로 검토하고 학습 방향을 가다듬는다. 사용자의 현재 이해도, 목표, 시간 제약을 반영하여 최적의 학습 경로를 설계한다.

## 트리거

- `/study:bs` — draft/refined 상태인 학습 계획서 목록을 보여주고 선택
- `/study:bs 1` — 1번 학습 계획서를 바로 브레인스토밍 시작
- 자연어: "스터디 방향 잡자", "학습 범위 조정하자", "스터디 브레인스토밍", "어디까지 공부할지 정하자"

## 실행 단계

### 1. 학습 계획서 선택

1. `study/docs/` 디렉토리를 스캔하여 학습 계획서 목록을 확인 (Glob 도구)
2. 각 계획서의 frontmatter에서 status를 읽어 상태 확인 (Read 도구)

- **번호가 주어진 경우**: 해당 계획서를 바로 열기
- **번호가 없는 경우**: 계획서 목록을 상태와 함께 표시하고 선택 요청

출력 예시:
```
| # | 주제 | 상태 | 난이도 | 최종 수정 |
|---|------|------|--------|----------|
| 1 | AWS 네트워크 인프라 | draft | intermediate | 2026-04-16 |
| 2 | Kafka 내부 동작 원리 | refined | advanced | 2026-04-15 |
```

### 2. 사용자 프로필 확인

메모리에서 사용자 정보를 로드:
- 현재 역할/경험 수준
- 관련 기술 스택 경험
- 이전 학습 이력 (있으면)

**첫 브레인스토밍 시 반드시 확인할 것**:
- "이 주제에 대해 현재 어느 정도 알고 계세요?" (기초/중급/고급)
- 이 정보로 학습 로드맵의 시작점과 깊이를 조절

### 3. 계획서 분석 및 대화 시작

선택된 학습 계획서를 읽고, 사용자에게 다음을 제시:

1. **현재 학습 계획 요약** — 주제, 목표, 로드맵을 3-5줄로 요약
2. **미결 사항** — 섹션 7의 미결 사항 목록
3. **첫 번째 질문** — 학습 방향에 가장 큰 영향을 미치는 **한 가지** 질문

### 4. 핑퐁 브레인스토밍

대화 원칙:
- **한 번에 한 가지 질문만** — 여러 질문을 한꺼번에 던지지 않는다
- **가능하면 선택지 제공** — "A vs B 중 어느 방향?" 형태로 구체적 선택지 제시
- **사용자 답변을 즉시 계획서에 반영** — 답변 받을 때마다 해당 섹션 업데이트
- **경험 수준 맞춤** — 사용자가 이미 아는 부분은 스킵 가능하도록 제안
- **사용자가 "됐어", "이정도면 됐다", "완료" 등 종료 의사를 밝히면** 브레인스토밍 종료

논의 진행 흐름:

#### Phase A: 학습 범위 정의
- 이 주제에서 어디까지 깊게 들어갈지 (개념 이해 vs 실무 적용 vs 면접 대비 vs 전체)
- 시간 제약: 몇 시간/며칠 투자 가능한지
- 우선순위: 로드맵 Phase 중 어디에 집중할지

#### Phase B: 학습 깊이 조절
- 기본 개념 중 스킵 가능한 것 식별 (이미 아는 내용)
- 심화 주제 중 꼭 포함할 것 선별
- 코드베이스 탐색 범위 (전체 스캔 vs 특정 모듈만)

#### Phase C: 면접 대비 방향 (해당 시)
- 목표 포지션/회사 특성 (있으면)
- 예상 면접 유형 (기술 면접, 시스템 설계, 코딩 테스트)
- 꼬리 질문 깊이: "왜?"를 몇 단계까지 파고들지

#### Phase D: 학습 전략
- 학습 방식 선호 (읽기 위주 vs 실습 위주)
- 출력 형태: 노트 정리 vs Q&A 카드 vs 두 가지 모두
- 코드 실습 포함 여부

### 5. 학습 계획서 업데이트

브레인스토밍 중/후에 학습 계획서 파일을 업데이트:

- 논의 결과를 해당 섹션에 반영 (Edit 도구)
- frontmatter의 `updated` 날짜 갱신
- `difficulty`, `estimated-hours` 조정 (논의 결과 반영)
- 학습 로드맵의 각 Phase에 구체적 소주제 추가/삭제/재배치
- 충분히 가다듬어졌으면 `status: draft` → `status: refined` 변경
- 사용자가 학습 시작 준비 완료를 선언하면 `status: refined` → `status: ready` 변경
- **상태 변경 시 `study/temp.md`의 학습 현황 표도 동기화**

### 6. 브레인스토밍 종료

종료 시 사용자에게:
- 변경된 내용 요약 (이번 세션에서 결정된 사항)
- 현재 학습 계획서 상태 (draft / refined / ready)
- 조정된 학습 시간 및 난이도
- 다음 단계 안내:
  - `refined`: "추가 조정이 필요하면 `/study:bs {번호}` 재호출"
  - `ready`: "`/study:exec {번호}` 로 학습 실행 가능"

## 학습 계획서 상태 흐름

```
draft → refined → ready → in-progress → completed
 (init)   (bs)      (bs)     (exec)        (exec)
```

- **draft**: init으로 1차 생성된 상태
- **refined**: 1회 이상 브레인스토밍으로 방향이 잡힌 상태
- **ready**: 학습 실행 가능한 수준으로 확정된 상태
- **in-progress**: 학습 실행 중
- **completed**: 학습 완료

## 대화 스타일

- **멘토 톤** — 학습을 돕는 시니어 동료처럼
- 사용자의 현재 수준을 존중하면서도 한 단계 위 도전을 권유
- "이건 이미 아시죠?" 보다는 "혹시 이 부분은 복습이 필요할까요?"
- 면접 관점에서 "이걸 물어보면 어떻게 답하실 건가요?" 식의 사전 점검도 자연스럽게 포함
- 기술적으로 더 효율적인 학습 경로가 있으면 적극 제안

## 에러 처리

- 학습 계획서가 없는 번호 지정 시: `/study:init {번호}` 로 먼저 생성하라고 안내
- `study/docs/` 가 비어있는 경우: `/study:init` 먼저 실행 안내
