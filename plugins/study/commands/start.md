# Study — Start (본격 심화 학습)

주제의 특정 소주제를 **면접 꼬리질문 수준 + 프로덕션 장애 대응 수준**까지 파고든다.

## 역할

`exec` 가 주제 전반의 breadth 라면, `start` 는 하나의 소주제에 대한 depth.

| 비교 | exec (preview) | start (심화) |
|---|---|---|
| 산출물 | `preview.md` 1개 | `{NN}-{subtopic}.md` N개 |
| 범위 | 주제 전체 소주제 지도 | 단일 소주제 하나 |
| 깊이 | 각 1-2문단 개요 | 내부 메커니즘 + 장애 + 면접 꼬리질문 |
| 분량 | 짧음 (~1000줄 이내) | 파일당 400-800줄 |

## 트리거

- `/study:start` — 심화 가능한 소주제 목록 표시 후 선택
- `/study:start 1` — 1번 주제의 소주제 목록 표시 (preview 기반)
- `/study:start 1 vpc` — 1번 주제의 `vpc` 소주제 심화
- `/study:start 1 --all` — 1번 주제의 모든 소주제 일괄 심화 (대용량)
- 자연어: "본격 스터디 시작", "VPC 깊이 파보자", "심화 학습"

## 산출물

**파일 경로**: `study/docs/{N}-{slug}/{NN}-{subtopic-slug}.md`

NN 은 2자리 (`01`, `02`, ...). subtopic-slug 은 kebab-case.

**템플릿**:

```markdown
---
parent: {N}-{parent-slug}
subtopic: {subtopic-slug}
index: {NN}
created: {YYYY-MM-DD}
level: deep
---

# {소주제 이름} — Deep Study

> 상위 주제: `plan.md` / 프리뷰: `preview.md`

## 1. 재확인 (프리뷰 요약)

preview.md 의 해당 소주제 요약을 2-3문장으로 복기.

## 2. 내부 메커니즘 (구현 레벨)

- 동작 원리를 Phase 1 보다 한 단계 아래로 분해
- 다이어그램, 자료구조, 알고리즘 수준
- 필요 시 실제 소스/스펙 인용 (RFC, AWS 공식 문서 등)

## 3. 설계 결정과 트레이드오프

- 왜 이렇게 설계되었는가
- 대안과 비교 (X 대신 Y 를 쓰는 이유)
- 성능/비용/복잡도 트레이드오프

## 4. 실제 코드/설정 예시

- msa 프로젝트의 적용 지점 (파일:라인 명시)
- 오픈소스 / 공식 문서의 전형 예시
- Terraform / K8s YAML / 코드 스니펫

## 5. 장애 시나리오 3개

각 시나리오:
- **증상**: 운영 현장에서 보이는 징후
- **진단**: 어떻게 원인을 좁혀가는가 (로그/메트릭/명령어)
- **해결**: 근본 원인 해결 + 재발 방지

## 6. 면접 꼬리 질문 트리 (3-4단계)

```
Q1: {기본 질문}
├── Q1-1: {꼬리 1}
│   └── Q1-1-1: {꼬리 2}
│       └── Q1-1-1-1: {꼬리 3}
└── Q1-2: {다른 방향 꼬리}
    └── Q1-2-1: {꼬리 2}
```

각 질문에 **핵심 답변 + 설명 + 실무 연결 (msa 기반)** 구조로 모범 답변.

## 7. 함정 질문 / 오해 포인트

- 흔히 잘못 알고 있는 사실
- 면접관이 의도적으로 틀린 전제를 깔고 물을 수 있는 것
- "~라고 답하기 쉬운데 실제로는 ~"

## 8. 자가 점검 체크리스트 (10개)

1. [ ] {핵심 질문 1}
2. [ ] {핵심 질문 2}
...

## 9. 참고 자료

- 공식 문서 링크
- 참고할 만한 글/영상
- 관련 오픈소스 코드
```

## 실행 단계

### 0. 준비

1. `study/docs/{N}-{slug}/plan.md` 와 `preview.md` 읽기
   - preview.md 가 없으면 `/study:exec {N}` 먼저 실행 안내
2. preview.md 에서 소주제 목록 추출
3. 사용자가 지정한 subtopic slug 확인 (없으면 목록 표시)

### 1. 기존 심화 파일 확인

- `study/docs/{N}-{slug}/{NN}-{subtopic-slug}.md` 이미 있으면 덮어쓰기/이어쓰기 선택
- 없으면 다음 index 번호로 신규 생성

### 2. 심화 학습 수행

- preview.md 의 해당 소주제 내용을 기반으로 깊이 확장
- 필요 시 context7 MCP 로 최신 공식 문서 참조
- 필요 시 Grep/Glob 으로 msa 코드베이스 탐색
- 필요 시 Explore 에이전트 위임 (광범위 탐색)

**작성 원칙**:
- **Depth 우선**: 같은 설명을 3번 다르게 말하기. 비유 → 메커니즘 → 코드.
- **꼬리 질문 필수**: 3단계 이상. 면접의 핵심.
- **msa 연결**: 가능한 모든 섹션에서 msa 프로젝트 실제 파일/ADR 참조.
- **파일당 400-800줄**: 너무 짧으면 얕음, 너무 길면 집중력 저하.

### 3. 파일 작성

템플릿 따라 1개 파일 작성.

### 4. preview.md 업데이트

preview.md 에 "심화 완료" 마커 추가:

```markdown
#### 1. VPC ✅ [심화: 01-vpc.md](01-vpc.md)
```

### 5. 결과 보고

- 생성된 파일 경로
- 커버된 핵심 꼬리 질문 수
- 다음 추천 심화 대상 (아직 완료 안 된 것 중 중요도 높은 것)

### 6. 일괄 처리 (`--all`)

- preview.md 의 모든 소주제를 순차 심화
- 각 파일 생성 후 preview.md 업데이트
- 완료 후 전체 진행 보고

### 7. 완료 처리

모든 소주제가 심화되면:
- `plan.md` 의 status 를 `completed` 로 변경
- `study/temp.md` 학습 현황 표 업데이트
- 최종 리포트: 심화 파일 N개, 총 라인 수, 면접 준비도 자가 평가 기준

## 작성 원칙 (중요)

- **Depth 측정 기준**: 이 파일 하나로 해당 소주제에 대해 3-4단계 꼬리 질문까지 방어 가능해야 함
- **중복 허용**: preview.md 와 일부 겹쳐도 OK. 심화는 반복 노출로 기억 강화.
- **실무 스토리**: 섹션 4-5 에서 msa 프로젝트 경험을 면접 답변으로 변환하는 문구 예시 포함
- **코드 인용**: msa 파일:line 또는 오픈소스 링크. "코드가 이렇게 생겼다" 수준.
- **비유 연결**: preview.md 의 핵심 비유를 각 심화 파일에서도 재활용

## 에러 처리

- `preview.md` 없음: `/study:exec {N}` 먼저 실행 안내
- 소주제 slug 가 preview 에 없음: 가용 slug 목록 출력
- 이미 파일 존재: 덮어쓰기/append 선택

## 상태 흐름

```
draft (init)
  → refined (bs)
  → ready (bs)
  → in-progress preview (exec)
  → in-progress deep (start 첫 실행)
  → completed (모든 소주제 start 완료)
```
